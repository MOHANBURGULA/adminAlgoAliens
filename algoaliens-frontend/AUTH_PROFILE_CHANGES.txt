Auth + Profile System Changes
Date: 2026-03-22

Overview
- Fixed login/signup redirect persistence and removed the auth loop on public auth pages.
- Fixed NextAuth Google route so /api/auth/providers and /api/auth/session return JSON correctly.
- Fixed password reset flow to use JWT + Nodemailer.
- Removed old-password-based password change flow.
- Fixed profile setup redirect logic so existing users do not see setup repeatedly.
- Fixed profile fetch to remain user-specific and dynamic.
- Fixed stale/static logged-in user state by clearing old sessions before every new manual or Google login.

Existing API routes and what changed

1. POST /api/auth/signup
- Route path unchanged.
- Frontend stores the returned JWT token and then validates user state using GET /api/users/me.
- Frontend checks GET /api/users/profile to decide whether to go to Dashboard or Profile Setup.
- Previous session data is cleared before a new signup starts.

2. POST /api/auth/login
- Route path unchanged.
- Frontend stores the returned JWT token and then validates user state using GET /api/users/me.
- Frontend checks GET /api/users/profile to decide whether to go to Dashboard or Profile Setup.
- Sign-in page auto-checks an existing token on load and redirects authenticated users away from the login screen.
- Previous session data is cleared before a new login starts.

3. POST /api/auth/change-password
- Route path unchanged.
- Logic changed completely.
- No longer accepts or validates oldPassword from the frontend.
- Logged-in user is extracted from JWT using req.user.
- Backend fetches the registered DB user using req.user.id.
- Backend generates a JWT reset token with 15 minute expiry.
- Backend sends reset email only to the registered email stored in DB.
- Works for both normal users and Google users.
- Frontend now sends this request only to trigger reset email delivery.
- Frontend shows visible in-app success/error status after the request.
- Direct browser GET access to /api/auth/change-password now returns a clear usage message instead of a confusing 404.

4. GET /api/users/me
- Route path unchanged.
- Used after login and signup to fetch the current logged-in user.
- Used on app load to validate persisted JWT and keep protected routes authenticated after refresh.
- Used as the source of truth for logged-in user data instead of trusting stale localStorage user state.

5. GET /api/users/profile
- Route path unchanged.
- Backend returns actual profile when found.
- Backend returns null when profile does not exist.
- Frontend uses this to decide:
  Dashboard if profile.skillLevel exists
  Profile Setup if profile is null or incomplete
- Profile and dashboard pages use this route to render user-specific profile data dynamically.

Additional internal handler

6. POST /api/auth/reset-password
- Added as internal password reset completion handler.
- Verifies reset JWT token.
- Extracts userId from token.
- Fetches user from DB.
- Hashes new password with bcrypt.
- Saves hashed password to DB.
- Returns success message.

Mail service changes
- Added reusable Nodemailer mail service.
- Reset email is sent only to the email retrieved from the database.
- Reset link format:
  http://localhost:3000/reset-password/{token}
- SMTP is configuration-based.
- Supports SMTP service-based config such as Gmail.
- Supports both key formats:
  SMTP_USER / SMTP_PASS
  EMAIL_USER / EMAIL_PASS
- If SMTP credentials are missing, backend returns an explicit error and frontend displays it in-app.

JWT auth contract alignment update
Date: 2026-03-23

Overview
- Aligned manual auth handling with the backend contract:
  POST /api/auth/signup returns { token, user }
  POST /api/auth/login returns { token, user }
  Authorization header must be Bearer <token>
- Preserved the existing Google OAuth flow and only aligned its session storage and redirect behavior with the JWT flow.
- Fixed admin login Unauthorized issues caused by split token storage and inconsistent frontend auth state.
- Fixed profile update so email is visible but not editable and is never sent in PUT /api/users/me.

Change details

1. Frontend session storage
- Frontend now treats localStorage token and user as the single source of truth for both student and admin sessions.
- Login, signup, admin login, and Google auth success all store:
  localStorage["token"]
  localStorage["user"]
- Legacy adminToken/adminUser values are cleared when a new auth session is stored.

2. Frontend request wrapper
- Shared axios wrapper now always reads localStorage["token"].
- All authenticated requests now send:
  Authorization: Bearer <token>
- Added 401 handling that clears auth storage and redirects to the correct login page.
- Added debug logs for:
  login response
  token storage
  Authorization headers sent with requests

3. Manual login flow
- Frontend login request uses POST /api/auth/login with JSON body:
  { email, password }
- Added preventDefault, loading state, inline error state, and awaited API submission.
- On successful login:
  admin role -> /admin/dashboard
  non-admin with profile -> /dashboard
  non-admin without profile -> /profile-setup

4. Manual signup flow
- Frontend signup request uses POST /api/auth/signup with JSON body:
  { name, email, password }
- Added preventDefault, loading state, inline error state, and awaited API submission.
- On successful signup:
  store token
  store user
  redirect to /profile-setup

5. Google OAuth compatibility
- Existing NextAuth Google OAuth flow remains unchanged.
- /auth/success still exchanges Google identity with the backend at POST /api/auth/google.
- Google-authenticated users now store the returned backend JWT in the same token/user keys used by manual auth.
- Backend manual login remains compatible with Google-created accounts by bypassing password comparison for records created through Google auth.

6. Admin access flow
- Admin login now validates role from response.user.role === "admin" before redirecting.
- Admin route bootstrap now validates the stored JWT through GET /api/users/me instead of relying on stale separate admin storage.
- Admin-only backend guard behavior remains unchanged and still requires user.role === "admin".
- Added backend guard failure logs to make Forbidden/Unauthorized cases easier to trace.

7. Profile update behavior
- Profile page email field is now disabled/readOnly.
- Frontend no longer sends email in PUT /api/users/me.
- Backend PUT /api/users/me now ignores email even if provided and only updates allowed fields.
- Profile save now refetches GET /api/users/me and GET /api/users/profile after update so UI reflects persisted backend state.

8. Backend login behavior
- Backend login continues to use bcrypt.compare() for normal manual-password users.
- Backend login now normalizes email before lookup.
- Backend login adds explicit success/failure logs for easier debugging.
- Existing Google OAuth route was not altered, preserving the working feature.

- Frontend TypeScript check passed with npx.cmd tsc --noEmit.
- Full frontend lint still reports unrelated pre-existing issues outside this auth change set.
